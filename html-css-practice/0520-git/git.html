什麼是 Pull Request (PR)？

在專業的企業開發流程與開源社群中，你不能直接 push 到 main/master 分支！
所有的代碼變更都必須透過建立 Pull Request 來提報，邀請團隊的其他成員進行 Code Review (代碼審查)，確認品質無誤後，才能合併至主分支。

程式碼品質控管：防範 Bug
團隊知識共享：相互學習
保持主線穩定：自動化測試整合
歷史紀錄清晰：利於日後追溯

PR 標準協作流程 (九大步驟)

1

步驟 1：同步主分支並建立功能分支

在開始任何開發之前，請務必先將本地的 main 分支與遠端倉庫同步，以防與他人衝突。

指令範例

git checkout main
git pull origin main
git checkout -b feature/team-page
2

步驟 2：開發功能網頁與設計

依據需求進行開發。例如此處我們新增了 team.html 及其專屬 CSS 樣式，並確認排版與響應式效果正常。

3

步驟 3：暫存並提交變更 (Commit)

使用 Conventional Commits 格式撰寫詳細的 Commit 訊息，說明修改的細節與原因。

指令範例

git add .
git commit -m "feat: 新增團隊介紹頁面

- 建立 team.html 展示公司核心團隊成員
- 新增團隊專用 CSS 樣式與 hover 卡片效果
- 更新所有頁面導航選單的連結
- 包含企業核心價值展示區"
4

步驟 4：推送分支到 GitHub

將你在本地新增的功能分支推送至 GitHub 遠端倉庫，以供建立 PR。

指令範例

git push origin feature/team-page
5

步驟 5：在 GitHub 建立 Pull Request

在瀏覽器前往該 repository，點擊「Compare & pull request」按鈕，填寫 PR 標題與下方提供的 Markdown 格式描述範本並送出。

6

步驟 6：進行 Code Review 意見修正

同事在 PR 的代碼行留言提出優化意見（例如補充圖片 alt 屬性）。直接在本地修改並再次推送至同分支，PR 頁面會自動更新。

指令範例

# 依建議修改代碼後暫存並 commit
git add .
git commit -m "style: 依據 Review 建議加上圖片 alt 屬性與優化邊距"

# 再次推送，PR 會自動同步更新
git push origin feature/team-page
7

步驟 7：合併 Pull Request

獲得同事核准 (Approve) 且 CI 測試通過後，點擊「Merge pull request」將程式碼合併至 main。隨即點擊「Delete branch」清理遠端功能分支。

8

步驟 8：同步本地端並刪除分支

切換回本地 main 分支，拉取合併後的最新主線，並安全刪除已完成使命的本地功能分支。

指令範例

git checkout main
git pull origin main
git branch -d feature/team-page
git fetch -p
9

步驟 9：檢查最終歷史線圖

使用圖形化指令檢查本地主分支是否包含最新 PR 合併的 Commit 節點。

指令範例

git log --oneline --graph -5

PR 標題最佳實踐 (Conventional Commits)

✅ 推薦寫法 (類型明確)
  • feat: 新增團隊介紹頁面功能
  • fix: 修復聯絡表單驗證錯誤
  • style: 改善響應式設計效果
❌ 糟糕寫法 (模糊不清)
  • 更新檔案
  • 修改東西
  • test

Code Review 審查五大重點

  • 1.
    功能性 (Functionality) 程式是否如預期正常運作？是否能正確解決目標問題？
  • 2.
    可讀性與維護性 (Readability) 程式碼是否乾淨容易理解？命名是否具備良好語意？
  • 3.
    一致性 (Style Consistency) 縮排、格式、架構是否符合團隊的 coding 規範與風格？
  • 4.
    安全性 (Security) 是否存在潛在的安全漏洞，如 SQL 注入、XSS、敏感資訊洩漏等？
  • 5.
    效能 (Performance) 是否有不必要的冗餘迴圈？是否存在可優化的設計？

企業協作：開發者典型的一天

這是大廠或專業研發團隊每天的日常開發節奏，保持此習慣可讓程式分支與協作井然有序：

🌅 1. 早上：同步環境

一早到公司，切到 main 分支，拉取遠端最新代碼並清理本地垃圾分支。

💻 2. 白天：功能開發與 PR

開功能分支進行開發、本地測試，完成後推上遠端並建立 PR 供同事審核。

🌆 3. 傍晚：修正、合併與清理

依 Review 建議修正代碼。確認無誤後將分支合併入 main，並刪除該分支。
